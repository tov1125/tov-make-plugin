# Agent Team Architect

Multi-agent 팀을 **분석 → 수집 → 계획 → 구현** 4단계를 거쳐 설계하고 코드 스캐폴딩까지 생성하는 스킬.
Anthropic의 harness design 원칙에 기반하며, 각 단계를 충실히 수행한 후 다음 단계로 넘어간다.

## 핵심 설계 철학

**"가장 단순한 구조에서 시작하고, 필요할 때만 복잡도를 추가하라."**

모든 harness 컴포넌트는 "모델이 혼자서는 못하는 것"에 대한 가정을 인코딩한다.
그 가정이 틀렸다면 해당 컴포넌트는 오버헤드일 뿐이다.
설계 전체에 걸쳐 항상 물어야 할 질문: **"이 agent를 빼면 어떤 문제가 생기는가?"**

**"하네스 조합의 가능성은 모델이 발전해도 줄어들지 않는다. 대신 이동한다."**
모델이 좋아지면 이전에 필요했던 구조가 불필요해지지만, 동시에 이전에는 불가능했던 새로운 조합이 가능해진다. 흥미로운 작업은 다음 새로운 조합을 찾는 것이다.

### 비용/시간 트레이드오프 인식

복잡도를 높이면 품질이 올라가지만 비용과 시간도 크게 증가한다. Anthropic의 실험에서:
- **단일 agent**: ~20분, ~$9 → 기본 기능은 동작하나 품질 불균일, edge case 미처리
- **풀 harness (3-agent)**: ~4-6시간, ~$125-200 → 완성도 높은 결과물, 구체적 버그 포착

사용자에게 이 트레이드오프를 설명하고, 태스크의 중요도/반복 빈도에 맞는 복잡도를 함께 결정할 것.

## 워크플로우 개요

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  분석    │ →  │  수집    │ →  │  계획    │ →  │  구현    │ →  │  검증    │
│ Analysis │    │ Collect  │    │  Plan    │    │  Build   │    │ Verify   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
 도메인 이해     레퍼런스 확보    아키텍처 설계   문서+코드 생성    품질 확인
```

각 단계는 이전 단계의 산출물을 기반으로 진행된다. 단계를 건너뛰지 말 것.
기존 시스템 개선의 경우에도 동일한 4단계를 따르되, 분석 단계에서 현재 시스템의 문제점 진단에 초점을 맞춘다.

## Step 1: 분석 (Analysis)

**목적:** 도메인과 태스크의 본질을 이해하여, 어떤 agent 구조가 필요한지 판단하기 위한 근거를 확보한다.

사용자와 대화하면서 자연스럽게 파악할 것. 모든 질문을 한 번에 던지지 않는다.

### 1-1. 태스크 특성 파악

- **무엇을 하는가?** — 앱 개발, 데이터 분석, 콘텐츠 생성, 연구 등 도메인 확인
- **결과물의 성격** — 객관적 검증 가능(코드, 데이터)인가, 주관적 판단 필요(디자인, 글)인가?
- **실행 규모** — 분 단위의 짧은 태스크인가, 시간 단위의 장기 태스크인가?
- **반복 빈도** — 일회성인가, 반복적으로 수행하는 파이프라인인가?

### 1-2. 품질 요구사항 파악

- 사용자가 **"잘 됐다"**고 판단하는 구체적 기준은 무엇인가?
- 가장 흔한 **실패 모드**는 무엇인가? (품질 불균일, 환각, context 소진, 창의성 부족 등)
- 결과물에서 모델이 **자연스럽게 잘하는 것**과 **약한 것**은 무엇인가?

### 1-3. 기존 시스템 진단 (개선의 경우)

기존 multi-agent 시스템을 개선할 때는 현재 문제를 먼저 진단한다:

| 증상 | 가능한 원인 | 개선 방향 |
|------|------------|-----------|
| 결과물이 뻔하고 템플릿 냄새 | 자기 평가 편향, 평가 기준 부재 | Generator-Evaluator 분리, 독창성 기준 추가 |
| 장시간 실행 시 품질 저하 | Context 소진, context anxiety | Context reset 또는 파일 기반 통신 도입 |
| 구조가 복잡한데 효과 미미 | 모델 능력 향상으로 불필요해진 컴포넌트 | 불필요 컴포넌트 제거 |
| Evaluator가 문제를 놓침 | Evaluator 프롬프트 튜닝 부족 | 엄격한 평가 기준 + 구체적 실패 예시 추가 |
| 스펙대로 구현이 안 됨 | 과도하게 상세한 스펙의 cascading error | Planner 추상화 수준 조정 |

### 1-4. 분석 산출물

분석 단계의 결과를 정리하여 사용자와 공유한다:

```markdown
## 분석 요약
- **도메인**: (예: 풀스택 웹앱 개발)
- **태스크 유형**: (예: 복잡한 장기 태스크, 주관적+객관적 혼합)
- **핵심 품질 요구**: (예: 기능 완성도 > UI 독창성 > 코드 품질)
- **주요 위험 요소**: (예: context 소진, 자기 평가 편향)
- **복잡도 판단**: 단일 agent / Generator+Evaluator / Planner+Generator+Evaluator
```

사용자가 분석 요약에 동의한 후 다음 단계로 진행한다.

## Step 2: 수집 (Collection)

**목적:** 설계에 필요한 레퍼런스, 패턴, 도메인 지식을 확보한다. 이 단계를 충실히 수행하면 계획 단계에서 근거 없는 추측을 줄일 수 있다.

### 2-1. 도메인별 레퍼런스 수집

`references/domain-guides.md`에서 해당 도메인의 가이드를 읽는다. 다음 항목을 확인:

- 해당 도메인에서 **권장하는 agent 구성**
- **평가 기준 예시** (그대로 쓰지 말고, 태스크에 맞게 조정할 참조용)
- 도메인 **특이사항** (주의할 점, 효과적인 패턴)

### 2-2. 기술 스택 및 도구 조사

사용자의 환경에서 사용 가능한 것을 파악한다:

- **사용 가능한 프레임워크**: Claude Agent SDK, LangGraph, CrewAI, AutoGen 등
- **사용 가능한 도구/MCP**: Playwright, 파일 시스템, API, DB 접근 등
- **제약 사항**: API rate limit, 비용 제한, 실행 환경 제약 등

### 2-3. 유사 사례 조사

비슷한 태스크에서 효과적이었던 패턴을 파악한다:

- 사용자가 이전에 구축한 시스템이 있는가?
- 프로젝트 내 관련 코드나 설정이 있는가?
- 참고할 외부 아키텍처나 문서가 있는가?

필요시 웹 검색이나 문서 조회로 추가 레퍼런스를 확보한다.

### 2-4. 수집 산출물

```markdown
## 수집 요약
- **참조한 도메인 가이드**: (예: 소프트웨어 개발)
- **선택한 프레임워크**: (예: Claude Agent SDK / 프레임워크 무관 범용)
- **사용 가능한 도구**: (예: Playwright MCP, 파일 시스템, GitHub API)
- **참고 사례**: (있으면 기술)
- **제약 사항**: (있으면 기술)
```

## Step 3: 계획 (Plan)

**목적:** 분석과 수집 결과를 종합하여 구체적인 아키텍처를 설계한다. 이 단계에서 모든 설계 결정의 근거를 명시한다.

### 3-1. 복잡도 결정

분석 결과를 기반으로 적절한 복잡도를 결정한다:

```
태스크를 단일 agent로 처리할 수 있는가?
├─ YES → 단일 agent 사용. 끝.
└─ NO → 자기 평가만으로 품질을 보장할 수 있는가?
    ├─ YES → Generator + 자체 검증
    └─ NO → Generator-Evaluator 분리
        └─ 태스크가 복잡하여 스펙이 필요한가?
            ├─ YES → Planner + Generator + Evaluator
            └─ NO → Generator + Evaluator
```

새 모델이 나올 때마다 이 판단을 재검토할 것. 이전에 필요했던 컴포넌트가 오버헤드가 될 수 있다.

### 3-2. Agent 역할 설계

Generator-Evaluator 분리가 핵심 패턴이다. 자기 평가 편향 때문에, 작업을 수행하는 agent와 평가하는 agent를 분리하는 것이 거의 항상 효과적이다.

```
[Planner] → [Generator] → [Evaluator]
    ↑                          │
    └──────── feedback ────────┘
```

**Planner** (선택적 — 복잡한 태스크에만)
- 짧은 요청을 포괄적 스펙으로 확장
- 높은 수준의 기술 설계, 세부 구현은 Generator에 위임
- 주의: 너무 상세한 스펙은 cascading error 유발. 방향을 제시하되 구현 방법은 열어둘 것.

**Generator**
- 실제 작업 수행
- Evaluator 피드백 기반으로 수정 또는 방향 전환 결정
- 자체 검증 후 Evaluator에 핸드오프

**Evaluator**
- 정의된 기준으로 결과물 평가
- 구체적이고 실행 가능한 피드백 제공
- 초기에는 문제를 발견하고도 합리화하는 경향이 있으므로, 엄격한 기준과 구체적 실패 예시를 명시할 것

### 3-3. 평가 기준 설계 (Criteria as Steering)

평가 기준은 단순한 채점표가 아니다. **기준의 문구 자체가 결과물의 방향을 결정한다.** Evaluator 피드백이 시작되기 전에, 기준 문구만으로 이미 Generator의 출력 성격이 달라진다. 이것은 Anthropic의 실험에서 확인된 핵심 발견이다.

예시: 프론트엔드 디자인에서 "museum quality"라고 쓰면 특정 시각적 수렴으로 향하고, "playful"이라고 쓰면 완전히 다른 결과물이 나온다. 기준 문구에 사용하는 단어를 의도한 결과물의 성격에 맞게 신중하게 선택할 것.

| 기준 | 정의 | 가중치 | 검증 방법 |
|------|------|--------|-----------|
| (도메인에 맞게 정의) | 구체적 설명 | 높음/중간/낮음 | 자동/수동/하이브리드 |

**원칙:**
- 모델이 자연스럽게 잘하는 것보다, **약한 부분에 높은 가중치**를 둘 것
  - 예: 코드 생성에서 기술적 정확성은 자연스럽게 잘 하므로 낮은 가중치, 독창적 UX는 약하므로 높은 가중치
- 주관적 품질도 원칙 기반 기준으로 측정 가능하게 만들 수 있다
  - 예: "독창성" → "템플릿 기본값이 아닌 의도적 창작 결정이 보이는가? 보라색 그라디언트+흰색 카드 같은 AI 생성 냄새가 나지 않는가?"
- 기준은 **4개 이하**로 유지 — 많으면 초점이 흐려진다
- 각 기준에 **hard threshold**를 설정하면 Evaluator가 "전체적으로 괜찮다"는 식의 모호한 판정을 내리는 것을 방지할 수 있다

### 3-4. 통신 및 Context 전략

**파일 기반 통신을 기본으로 사용할 것.** Agent 간 정보 전달은 파일 시스템을 통해 이루어져야 장시간 세션에서도 coherence가 유지된다.

| 전략 | 장점 | 단점 | 적합한 경우 |
|------|------|------|-------------|
| **Context Reset** | 깨끗한 슬레이트, context anxiety 해소 | 오케스트레이션 복잡도 | 독립적 단계가 명확한 태스크 |
| **Compaction** | 연속성 유지 | context anxiety 미해결 | 연속적 흐름이 중요한 태스크 |
| **파일 기반 통신** | agent 간 coherence 유지 | 파일 관리 필요 | 모든 multi-agent 시스템에 권장 |

### 3-5. Sprint Contract (장시간 태스크용)

실행 시간이 1시간 이상인 태스크에서는 Generator와 Evaluator가 구현 전에 합의:

- 이번 스프린트에서 구현할 구체적 항목
- 테스트 가능한 성공 기준
- "완료"의 정의

### 3-6. 반복 전략

- 최대 반복 횟수 결정 (태스크 복잡도에 비례, 일반적으로 3-15회)
- 종료 조건 명확히 정의 (무한 루프 방지)
- Generator가 피드백 기반 **수정(refinement)** vs **방향 전환(pivot)** 을 전략적으로 선택하도록 설계

### 3-7. 계획 산출물

완성된 아키텍처 설계를 사용자에게 제시하고 승인을 받은 후 구현으로 넘어간다:

```markdown
## 아키텍처 계획
- **패턴**: (예: Planner + Generator + Evaluator)
- **각 Agent 역할 요약**: (한 줄씩)
- **평가 기준**: (표)
- **통신 방식**: (예: 파일 기반, workspace/ 디렉토리)
- **Context 전략**: (예: Context Reset + 파일 기반)
- **반복 전략**: (예: 최대 10회, Evaluator 점수 80% 이상이면 종료)
- **설계 근거**: 각 결정이 왜 이 선택인지 한 줄씩
```

## Step 4: 구현 (Build)

**목적:** 승인된 계획을 바탕으로 설계 문서와 실행 가능한 코드 스캐폴딩을 생성한다.

### 4-1. 설계 문서 생성

파일 위치는 사용자의 프로젝트 루트에 `docs/agent-architecture/` 디렉토리를 기본으로 한다.

#### `architecture.md` — 아키텍처 개요

```markdown
## 개요
- 목적: (한 문장)
- 패턴: (Generator-Evaluator / Planner-Generator-Evaluator / 커스텀)
- 예상 실행 시간:
- Context 관리 전략:

## Agent 구성

### [Agent 이름]
- **역할**:
- **입력**:
- **출력**:
- **사용 도구**:
- **성공 기준**:

(각 agent별 반복)

## 통신 흐름
(agent 간 데이터 흐름 다이어그램 — 텍스트 기반)

## 평가 기준
| 기준 | 정의 | 가중치 | 검증 방법 |
|------|------|--------|-----------|

## 반복 전략
- 최대 반복 횟수:
- 종료 조건:

```

#### `evaluation-criteria.md` — 상세 평가 기준 및 채점 루브릭

#### `contracts.md` — Sprint Contract 템플릿 (장시간 태스크만)

### 4-2. 코드 스캐폴딩 생성

프레임워크에 구애받지 않는 범용 구조를 기본으로 하되, 사용자가 특정 프레임워크를 지정하면 그에 맞춘다.

#### 기본 디렉토리 구조

```
agents/
├── orchestrator.py      # agent 간 흐름 제어, 반복 로직, 종료 조건
├── planner.py           # (필요시) 스펙 생성
├── generator.py         # 작업 수행 agent
├── evaluator.py         # 평가 agent
├── shared/
│   ├── context.py       # 파일 기반 context 관리
│   ├── criteria.py      # 평가 기준 정의 (설정 파일로 관리, 튜닝 용이)
│   └── contracts.py     # sprint contract 관리 (필요시)
├── prompts/
│   ├── planner.md       # planner system prompt
│   ├── generator.md     # generator system prompt
│   └── evaluator.md     # evaluator system prompt
└── workspace/           # agent 간 파일 기반 통신 디렉토리
```

#### 스캐폴딩 생성 원칙

- 각 agent는 독립적으로 실행 가능해야 한다
- Agent 간 통신은 `workspace/` 디렉토리의 파일을 통해 이루어진다
- System prompt는 별도 `.md` 파일로 분리하여 수정 용이하게 한다
- Evaluator의 평가 기준은 코드가 아닌 설정 파일로 관리한다 (튜닝 용이성)
- 오케스트레이터는 반복 로직과 종료 조건을 담당한다

### 4-3. Evaluator 프롬프트 작성 및 튜닝 가이드

Evaluator 튜닝은 시스템 전체 품질에 직접적 영향을 미친다. Anthropic의 경험에 따르면 **"Claude는 기본적으로 나쁜 QA agent이다."** 초기 Evaluator는 문제를 발견하고도 "전반적으로 괜찮다"고 합리화하며, 피상적으로만 테스트하고 edge case를 놓친다.

#### Evaluator 프롬프트 필수 패턴

- 발견한 문제를 "괜찮다"고 넘기지 말고 **반드시 보고**하도록 명시
- **문제를 먼저 모두 나열**한 후, 마지막에 점수를 매기도록 순서 지정 (점수를 먼저 매기면 이미 결정된 점수에 맞춰 문제를 축소하게 된다)
- 피상적 테스트가 아닌 **edge case까지** 확인하도록 지시
- 각 평가 기준에 **hard threshold**를 설정하여 모호한 "전체적으로 OK" 판정 방지

#### 구체적 실패 사례를 예시로 제공

Evaluator가 어떤 수준의 버그를 잡아야 하는지 구체적 사례로 보여줘야 한다. Anthropic의 게임 메이커 프로젝트에서 Evaluator가 포착한 실제 버그들:

- **로직 버그**: 사각형 채우기 도구가 드래그 끝점에만 타일을 배치하고 영역 전체를 채우지 않음. `fillRectangle` 함수가 존재했지만 mouseUp 이벤트에 연결되지 않음
- **조건 버그**: Delete 키 핸들러가 `selection`과 `selectedEntityId`를 모두 요구했지만, 엔티티 클릭 시 하나만 설정됨. 조건이 OR이어야 하는데 AND로 되어 있음
- **라우팅 버그**: `/frames/reorder` 경로가 `/{frame_id}` 뒤에 정의되어, FastAPI가 "reorder"를 정수 frame_id로 파싱하려다 422 에러 발생

이런 수준의 구체성이 Evaluator 프롬프트에 포함되어야 Evaluator가 표면적 확인을 넘어서 실제 동작 검증을 수행한다.

#### Evaluator 반복 튜닝 프로세스

Evaluator 프롬프트는 한 번에 완성되지 않는다. 아래 과정을 반복:

1. Evaluator 로그를 읽는다
2. Evaluator의 판단과 인간 판단 사이의 차이(divergence)를 찾는다
3. 해당 차이를 해소하도록 프롬프트를 수정한다
4. 다시 실행하여 개선 여부를 확인한다

## Step 5: 검증 (Verify)

**목적:** 완성된 설계와 스캐폴딩이 원래 요구사항을 충족하는지, 불필요한 복잡도가 없는지 확인한다.

### 5-1. Load-Bearing Assumption 검증

각 agent에 대해:
- [ ] 이 agent를 제거하면 구체적으로 어떤 문제가 발생하는가?
- [ ] 현재 모델이 이 역할을 단독으로 수행할 수 없다는 근거가 있는가?
- [ ] 이 agent가 추가하는 복잡도 대비 가치가 충분한가?

### 5-2. 설계 일관성 검증

- [ ] 모든 agent 간 입출력 인터페이스가 명확히 정의되어 있는가?
- [ ] 평가 기준이 태스크의 실제 품질 요구사항을 반영하는가?
- [ ] Context 관리 전략이 예상 실행 시간에 적합한가?
- [ ] 종료 조건이 명확하여 무한 루프 위험이 없는가?

### 5-3. 간소화 검증 (체계적 간소화 방법론)

급진적으로 여러 컴포넌트를 한 번에 제거하면, 어떤 것이 실제로 필요했는지 알 수 없다.
Anthropic의 교훈: **한 번에 하나의 컴포넌트만 제거하고 영향을 측정하라.**

- [ ] 불필요한 agent나 단계가 없는가? → 의심되는 컴포넌트를 하나씩 빼고 결과 비교
- [ ] Sprint contract가 필요 없는 짧은 태스크에 과도한 구조를 적용하지 않았는가?
- [ ] "나중에 필요할 수도 있으니까"로 추가한 컴포넌트가 없는가?
- [ ] 최신 모델에서 이전에 필요했던 구조가 여전히 필요한가? (예: Opus 4.6에서는 sprint 분해 없이 2시간+ 연속 빌드가 가능해짐)

### 5-4. 분석 단계와의 정합성

- [ ] 분석에서 파악한 핵심 품질 요구가 평가 기준에 반영되었는가?
- [ ] 분석에서 식별한 주요 위험 요소에 대한 대응이 설계에 포함되었는가?
- [ ] 수집한 레퍼런스와 제약 사항이 구현에 반영되었는가?

검증 결과 문제가 발견되면 해당 단계로 돌아가 수정한다.
